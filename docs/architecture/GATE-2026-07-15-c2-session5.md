## Blockers

- **New-file rollback удаляет файл, даже если его легитимно изменили после создания** — `app/workspace/snapshot.py:340-361`. Staleness-проверка явно пропускается для new-file snapshot'ов (`if snapshot.file_exists_before:` на строке 343 — блок с проверкой хэша целиком внутри этого `if`, для `file_exists_before=False` не выполняется вообще). Далее тело rollback (356-361) решает restore-vs-delete по `if snapshot.content:` — а `content` для new-file snapshot всегда `b""` (falsy), поэтому всегда идёт в `else: full.unlink()`, безусловно удаляя файл, если он существует. Собственный тест это не просто не ловит — **явно проверяет как корректное поведение**: `tests/test_workspace_snapshot.py:697-716` (`test_new_file_rollback_deletes_file`) создаёт snapshot нового файла, затем пишет в файл `"created content"` (эмулируя стороннее изменение — комментарий `# The file now exists (was just created)` описывает это как «этот же экшн создал», но по факту тест ничем не отличает «этот же агент дозаписал» от «другой агент/сессия записали после создания»), и утверждает, что rollback его удаляет (`assert not (project / "new_file.txt").exists()`). Ровно тот сценарий из чек-листа: «New-file rollback не удаляет файл, если его кто-то изменил после операции» — сейчас удаляет безусловно. Companion-тест `test_new_file_stale_check_skipped` (снапшот.py тесты, строки 396-413) прямым текстом документирует, что staleness-check «skipped» для new-file случая — то есть это не оплошность, а спроектированное (и неверное) поведение.
  - Смягчающее обстоятельство: `SnapshotStore.capture()` нигде не вызывается за пределами `app/workspace/snapshot.py` и его собственного теста (`grep -rn "\.capture(" app/` — 0 внешних вызовов) — то есть баг пока недостижим через REST/MCP, он живёт только в изолированном, ещё не подключённом модуле. Это снижает срочность немедленного инцидента, но не снимает статус BLOCKER для приёмки самого C2-компонента: если Agent 3/4 в следующей сессии подключит `capture()`/`rollback()` к реальному write-пути, доверяя тому, что модуль уже «протестирован и корректен», баг станет живым немедленно и без дополнительного ревью — тестовый набор сегодня *гарантирует* прохождение неверного поведения.

## Majors

- **Empty existing file не отличается от non-existing file** — `app/workspace/snapshot.py:141-146` (`_file_hash`). Функция возвращает `""` и для отсутствующего файла (`if not path.exists(): return ""`), и для существующего файла с пустым содержимым (`return _compute_hash(content) if content else ""` — falsy-content тоже даёт `""`). Snapshot, сделанный для существующего пустого файла, получает `expected_current_hash=""` — тот же сентинел, что и «файла не было». Сценарий: агент A снапшотит существующий пустой файл, кто-то удаляет этот файл целиком до rollback → `_file_hash` на удалённом файле тоже вернёт `""` → staleness-check **не заметит** разницы между «файл остался пустым» и «файл исчез», rollback пройдёт как обычный (не подняв `StaleSnapshotError`), а тело rollback (`if snapshot.content:` — falsy) уйдёт в `else: if full.exists(): unlink()` — файла уже нет, ничего не произойдёт, но `rolled_back=True` будет возвращено, хотя реального восстановления состояния (было пустое существование файла — стало отсутствие файла) не случилось. Не проверено ни одним тестом — ни `test_capture_empty_content_valid` (только new-file случай), ни staleness-тесты (используют непустой `hello.txt`).
- **`receipt_id` НЕ связывает receipt и snapshot** — вопреки собственному докстрингу `app/workspace/snapshot.py:14-16` («Snapshot.receipt_id is the linking key between snapshot ↔ ChangeReceipt»). `ChangeReceipt.receipt_id` генерируется в `app/workspace/receipts.py:17-19,45` (`_generate_receipt_id()`, префикс `rcpt_`); `Snapshot.receipt_id` генерируется независимо в `app/workspace/snapshot.py:272` (`f"r_{uuid.uuid4().hex[:16]}"`, префикс `r_`, другой формат). Ни один код-путь не передаёт один и тот же id в обе стороны — и не может, так как `receipts.make_receipt()` и `snapshot.SnapshotStore.capture()` сейчас не вызываются вместе ни в одном модуле (см. Blocker выше). Item 3 чек-листа («receipt_id связывает receipt и snapshot») сейчас не выполнен ни по факту, ни по формату ID.
- **`safe` по умолчанию `False`, спека требует `True`** — `app/workspace/edit.py` (`project_file_write`, `project_file_edit`, `project_apply_patch` — все три сигнатуры `safe: bool = False`) против спеки §4.2 («get a new optional parameter `safe: bool = True`»). Это прямо противоположно задуманной модели «safe by default, C1 caller opts out» — сейчас «unsafe by default, caller opts in». Новый implementation-matrix footnote¹ упоминает только то, что `safe=` «returns a simplified dict», но не упоминает, что направление default'а инвертировано относительно спеки — то есть сам матрица-документ не полностью truthful в этой детали, хоть и честен в остальном. Практического риска сегодня нет (ничего не wired в REST/MCP), но при вайринге, если кто-то скопирует сигнатуру как есть, поведение по умолчанию будет молча небезопасным.
- **CI по-прежнему не гоняет auth-boundary тест нигде** — `.github/workflows/ci.yml` не изменился в части исключения `host_smoke` (`pytest -m "not host_smoke"` как и раньше). `tests/test_mtls_e2e.py:19` теперь честно промаркирован `pytestmark = pytest.mark.host_smoke` (не тихое исчезновение — уже прогресс, зафиксировано и в implementation-matrix как известный gap #5), но в репозитории существует ровно один workflow-файл (`.github/workflows/ci.yml`) и ни одного `.gitea/workflows/*` — то есть mTLS/auth-boundary e2e тест не запускается ни в одном пайплайне, не только в этом. Это не регрессия от C2, но и не исправлено — открытый MAJOR gap, честно задокументированный, но не устранённый.

## Merge recommendation

**Не мержить `c2-spec-arch` в этом виде.** Конкретно перед следующим шагом (REST/MCP wiring, которое явно запланировано как следующая сессия) обязательно закрыть:

1. New-file rollback должен либо (а) require staleness-check тоже для new-file случая — сравнивать текущее состояние файла с «должен был появиться с конкретным `after_hash` от исходной операции», а не безусловно удалять; либо (b) явно задокументировать и заблокировать rollback, если файл был изменён кем-то ещё после создания (`current_hash` не совпадает с тем, что было записано операцией). Тест `test_new_file_rollback_deletes_file` должен быть переписан так, чтобы проверять отказ, а не удаление, в сценарии стороннего изменения.
2. Различить «пустой существующий файл» от «файла не существует» — не использовать один и тот же сентинел `""` для обоих состояний в `_file_hash`/`expected_current_hash` (например, `None` для «не существует», `""` только для «существует и пуст», либо отдельный boolean).
3. Либо реализовать реальную связку `receipt_id` между `ChangeReceipt` и `Snapshot` (единый ID, передаваемый в оба места одним вызовом), либо убрать это заявление из докстринга снапшота, пока связки нет.
4. Привести направление default'а `safe=` в соответствие со спекой (или явно и заметно — не только в сноске — задокументировать сознательное решение оставить `False` по умолчанию до полного вайринга).

Пункты 5 (CI host_smoke) и agent read-scopes — не блокируют именно этот C2-диф (они не регрессия от него), но остаются в открытом backlog и не должны считаться закрытыми при следующей самооценке архитектора.

Command-policy bypass matrix — намеренно не проверялась глубоко по инструкции; фиксирую как отдельный **C3 backlog/blocker-candidate**, не смешивая с этим gate.

### Нарушения — BLOCKER

- **[GOV-04 / SEC-02]** BLOCKER — `docs/superpowers/specs/2026-07-15-workspace-control-plane-phase-c2-safe-ide.md` (коммит `3ea5b21`, статус «Final — ADR decisions closed, ready for implementation»). Спека декларирует как принятые решения: persistent JSONL audit trail, memory-cap 64 MiB с LRU-вытеснением, `safe=True` по умолчанию с обратной совместимостью C1 при `safe=False`, staleness-защиту rollback, новые скоупы `workspace:snapshot`/`project:rollback`, эндпоинты `/preview /verify /rollback /snapshot /audit`. Ни один из этих пунктов не реализован и не подключён на HEAD: `app/routers/workspace.py` не содержит ни одного из перечисленных роутов; `VALID_AGENT_SCOPES` (`app/auth_middleware.py:56-68`) не содержит новых скоупов; параметр `safe` отсутствует в сигнатурах `write_file/edit_file/patch_file` (`app/workspace/edit.py:166-236`, `app/routers/workspace.py:344-403`) — он либо всегда возвращает новый формат ответа, либо не понимает флаг вовсе. Архитектор зафиксировал решение как готовое к продакшену, не сверив его с уже написанным на той же ветке кодом — нарушение GOV-04 (доказательства, а не «выглядит хорошо») и по сути SEC-02 (документ о защите не подтверждён фактической конфигурацией).
- **[BLOCKER, data-loss/security]** — `app/workspace/snapshot.py:237-298` (`SnapshotStore.rollback()`). Метод не сравнивает текущий хэш/mtime файла со снапшотом перед перезаписью — просто читает сохранённое содержимое и делает `_atomic_write` поверх текущего состояния. Спека этого же коммита (§3.5 п.3, §8.2, §8.7) требует отклонять rollback с `STALE_SNAPSHOT`, если файл был изменён после создания snapshot. Тест `tests/test_workspace_snapshot.py:207-225` (`test_rollback_restores_content`) явно симулирует внешнее изменение файла между snapshot и rollback (`# Simulate mutation`, `write_bytes(b"MODIFIED!")`) и **утверждает как корректное поведение**, что rollback тихо затирает эту модификацию. Сценарий: агент A правит файл → агент/сессия B легитимно правит тот же файл → агент A откатывает свой старый snapshot → правки B бесследно уничтожаются без предупреждения и без возможности восстановления (rollback ничего не версионирует поверх). Это ровно тот сценарий, который спека называет закрытым ADR-решением.
- **[TEST-15 / CI-06 / CI-10 / CI-11]** BLOCKER-кандидат — `pyproject.toml`, `.github/workflows/ci.yml` (`pytest -m "not host_smoke"` — единственный вызов pytest в единственном workflow). Маркер `host_smoke` полностью исключает из обязательного CI: весь `tests/test_mtls_e2e.py` (регрессионный тест на bypass mTLS/Authelia fallback, т.е. тест auth-границы) и весь `tests/test_workspace_tools.py` (9 read-тестов нового control plane), частично `tests/test_opencode_runner_wrapper.py`. Других workflow (в `.gitea/workflows`) не существует. Итог: ни integration-, ни smoke-уровень для auth-границы и для workspace-функциональности не гейтит merge — зелёный CI не является доказательством работоспособности именно тех путей, которые в этом аудите оказались сломаны (rollback staleness, scope-модель). Прямое проявление принципа §13: «зелёный pipeline не доказывает работоспособность сам по себе».
